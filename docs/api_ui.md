# Instructions for Claude Code — API Layer + React UI Integration

> Complements the Stage-1 PRD. This document defines the **FastAPI layer** that exposes the existing `src/ola/` learning system to a **React frontend**, and the **UI contract** the frontend builds against. Read alongside the PRD; do not duplicate or move any learning logic.

## 1. Core principle — the API is a THIN shell
All agent / memory / pipeline / KG logic stays exactly where the PRD puts it, inside the `src/ola/` package. The API layer **only**:
- receives HTTP requests,
- calls existing functions (`pipeline.process_interaction(...)`, `run_consolidation(...)`, profile/synopsis reads),
- serializes the returned Pydantic models to JSON.

**No business logic in route handlers. No learning logic in the frontend.** The frontend is a view plus a few buttons. If a route handler starts making decisions about memory, tiers, or personalization, that logic belongs in `src/ola/` instead.

## 2. Stack
- **FastAPI** + **uvicorn**. Async route handlers (the agents make async LLM calls).
- Reuse the existing **Pydantic v2** models as response models — they serialize to JSON for free; do not define parallel DTOs unless a response needs to *omit* fields (see §5 integrity rule).
- **CORS**: allow the React dev origin (e.g. `http://localhost:5173` for Vite or `http://localhost:3000` for CRA). Make the allowed origin an env var.
- New code lives in `src/ola/api/` (e.g. `app.py`, `routes.py`, `schemas.py`). The existing package is unchanged except where it already exposes the functions the API calls.

## 3. CRITICAL integrity rule — protect the evaluation
The simulated operator personas (`sim/`) have **hidden ground-truth traits**. The whole evaluation depends on those traits being invisible to the system under test — and "the system" now includes anything the browser can reach.

- **No endpoint may return a persona's ground-truth traits to the client** as part of normal operation.
- The frontend may **drive** a persona ("generate the next interaction") but must never **see the answer key**.
- Ground-truth traits are **never** an input to the learning pipeline — the pipeline only ever sees the observable interaction text.
- The persona's hidden traits stay server-side in `sim/`; the generator exposes only the produced interaction, not the traits behind it.
- **One deliberate exception:** a clearly-separated `GET /eval/{operator_id}` "reveal" endpoint (see §4) may return an inferred-vs-ground-truth comparison **for demonstration/scoring only**. It is computed server-side, is conceptually the answer key, and its output is never fed back into the learning system. Keep it obviously separate from the learning routes.

## 4. Endpoints

### `POST /interaction`
Submit one operator interaction (typed by the user, or generated by the persona) and run the hot path.

Request:
```json
{
  "operator_id": "op_A",
  "source": "user" | "simulated",
  "message": "Can you show me how to clear alarm 4471?"   // required if source=user; omit if simulated
}
```
Behaviour:
- `source: "simulated"` → server asks the persona to generate the next interaction, then processes it. Response includes what the persona "said" so the UI can display it as the operator's turn.
- `source: "user"` → process the provided `message`.

Response:
```json
{
  "interaction": {
    "id": "evt_123",
    "operator_message": "Can you show me how to clear alarm 4471?",
    "timestamp": "2026-06-23T09:14:00Z",
    "alarm_code": "4471",
    "shift": "day"
  },
  "assistant_reply": "Here are the visual steps for alarm 4471 ...",
  "validation_requested": false,
  "signals_extracted": [
    {"category": "INSTRUCTION_MODALITY", "value": "VISUAL", "observation": "asked for visual steps"}
  ],
  "memory_operations": [
    {"op_type": "REINFORCE", "target_item_id": "mem_007", "category": "INSTRUCTION_MODALITY"}
  ],
  "profile": { /* current profile, see §5 */ }
}
```
`signals_extracted` and `memory_operations` are included deliberately — they let the UI show *what the system learned from this turn* (a strong "show your work" transparency panel). `profile` is returned on every interaction so the live panel refreshes without a second call.

### `POST /shift/end`
The "End Shift" button. Runs consolidation (the slow loop): Reviewer consolidation, tier recompute, KG projection of newly-established items, synopsis regeneration. Returns before/after so the UI can show the contrast.

Request:
```json
{ "operator_id": "op_A", "shift": "day" }
```
Response:
```json
{
  "no_significant_updates": false,
  "changes": {
    "tier_transitions": [{"item_id": "mem_007", "from": "tentative", "to": "established"}],
    "new_items": [],
    "superseded": [{"item_id": "mem_003", "by": "mem_011"}]
  },
  "profile_before": { /* ... */ },
  "profile_after":  { /* ... */ },
  "synopsis_before": "Operator A, early observations ...",
  "synopsis_after":  "Operator A: confident with die-attach alarms; prefers visual guidance ..."
}
```
Must be **safe to press repeatedly and on an empty shift**: if nothing material accumulated since the last consolidation, return `no_significant_updates: true` with `profile_after == profile_before` and the synopsis unchanged. Do not fabricate changes or bump the synopsis version on a no-op.

### `GET /profile/{operator_id}`
Current tiered profile for the live panel. Shape in §5.

### `GET /synopsis/{operator_id}`
```json
{ "text": "Operator A: confident with ...", "generated_at": "2026-06-23T17:00:00Z", "version": 3 }
```

### `GET /operators`
List of available simulated operators for the picker. **No ground-truth traits.**
```json
{ "operators": [ {"id": "op_A", "name": "Operator A", "machine_type": "WireBond"} ] }
```

### `POST /reset/{operator_id}`  (demo utility)
Clears the operator's learned state (events, signals, memory operations, synopsis) so the learning arc can be re-run live. Does **not** touch the domain KG or the persona definition.

### `GET /eval/{operator_id}`  (optional — the "reveal", see §3)
Server-computed inferred-vs-ground-truth comparison, for the demo punchline only. Returns matched/missed/spurious inferred traits and a simple score. Clearly separated from learning routes; never an input to the pipeline.

## 5. Shared profile shape
Return the profile consistently everywhere it appears:
```json
{
  "operator_id": "op_A",
  "items": [
    {
      "id": "mem_007",
      "text": "Prefers visual, step-by-step instructions",
      "category": "INSTRUCTION_MODALITY",
      "status": "established",        // tentative | established | confirmed
      "evidence_count": 8,
      "source_event_ids": ["evt_12", "evt_19"],
      "last_updated": "2026-06-23T16:40:00Z"
    }
  ]
}
```
Only **active** items (status != superseded) appear by default. `status` and `evidence_count` are what the UI renders as the confidence tier — they come from the code-level tier rule, never from an LLM.

## 6. UI contract — three panels + two controls
The frontend's only job is to make the learning **visible**. Suggested layout:
1. **Conversation panel** — operator turns + assistant replies. Two input affordances: a free-text box (`source: user`) and a "Next simulated interaction" button (`source: simulated`). Optionally show the per-turn `signals_extracted` / `memory_operations` as a subtle "what was learned" annotation.
2. **Live profile panel** — the active memory items grouped by category, each with its tier badge (tentative / established / confirmed) and evidence count. Refreshes from the `profile` returned by each `/interaction` call.
3. **Synopsis panel** — the natural-language synopsis paragraph, with its version.

Controls:
- **End Shift** → `POST /shift/end`; on response, visibly animate or diff the before/after profile + synopsis so the contrast is obvious (this is the core "it learned over a shift" moment). Show "no significant updates this shift" on a no-op.
- **Operator picker** → `GET /operators`, switches `operator_id`.
- (Optional) **Reset** and **Reveal (eval)** for live demos.

## 7. Scope guardrails — do NOT gold-plate
The frontend is graded for making the concept legible, not for polish. **Do not build:** auth/login, user accounts, websocket streaming (plain request/response is fine), animations beyond the shift-end diff, mobile responsiveness, a design system, or persistence beyond what SQLite already gives. Three panels and two buttons that clearly show profile + synopsis evolving beats a polished app that hides the learning. If you find yourself adding UI features that don't surface the learning process, stop.

(Streaming the assistant reply token-by-token is the *one* optional nicety worth considering later; default to non-streaming.)

## 8. State & config
- State is per `operator_id`, backed by the existing SQLite store; it persists across server restarts (fine for the demo). `/reset` clears one operator.
- All model/provider config (model name, base URL, API key) stays in env vars as the PRD specifies — the API adds the CORS origin env var and nothing else security-sensitive.
- Consolidation remains **manual**, triggered only by `POST /shift/end`. Nothing in the hot path calls `run_consolidation`. State the production framing in a comment: in production this fires automatically at shift boundaries (scheduler / MES event); here it is a button so the demo can drive it.

## 9. Run instructions (include in README)
- Backend: `uvicorn ola.api.app:app --reload --port 8000` (with env vars set).
- Frontend: points at `http://localhost:8000`; CORS origin env var set to the frontend's dev URL.
- Document the env vars and the two start commands so a reviewer can run both halves.

## 10. Acceptance criteria
- React app talks to FastAPI; submitting an interaction (typed or simulated) returns an assistant reply and the updated profile, and the live panel reflects it without a manual refresh.
- "End Shift" runs consolidation and the UI visibly shows the before/after profile + synopsis contrast; pressing it on an empty shift is a clean no-op.
- No endpoint reachable by the client exposes a persona's ground-truth traits (except the explicitly-separated `/eval` reveal); ground truth is never passed into the pipeline.
- All learning logic remains in `src/ola/`; route handlers are thin pass-throughs.
- Backend and frontend each start with a single documented command.
